# Cat Catch AI 协作规范

本文件用于约束 AI 代理、自动化工具和协作开发者在本仓库中的工作方式，目标是：低风险改动、保持兼容性、减少无效重构、提高中文协作体验。

## 1. 交流与输出约定

- 默认使用中文交流、说明问题、提交变更摘要。
- 优先直接解决问题，避免只给方案不落地。
- 如果需求存在明显分支且影响较大，先说明选项和风险，再继续修改。
- 输出应简洁、可执行，避免泛泛而谈。

## 2. 项目性质

- 本项目是浏览器资源嗅探扩展，核心面向 Chrome/Edge Manifest V3。
- Firefox 通过单独的 `manifest.firefox.json` 兼容。
- 项目不是典型前端工程，没有打包器、没有 TypeScript、没有框架层。
- 代码以原生 HTML/CSS/JavaScript 为主，直接在源码目录运行和加载。

## 3. 目录与职责

- `manifest.json`：Chrome/Edge 扩展入口清单。
- `manifest.firefox.json`：Firefox 兼容清单。
- `js/background.js`：后台核心逻辑，负责抓包、缓存、消息分发、下载协同。
- `js/content-script.js`：内容脚本，负责页面媒体状态、页面桥接、页面内消息监听。
- `js/popup.js`：弹窗页主逻辑，负责资源列表渲染与交互。
- `js/options.js`：设置页逻辑。
- `js/m3u8.js`：m3u8 下载与解析页面主逻辑，体量较大，修改要谨慎。
- `js/init.js`：全局配置、默认项、兼容性兜底、脚本注册。
- `js/function.js`：公共方法与模板处理函数。
- `catch-script/`：注入页面的辅助脚本，如深度搜索、录制、抓取等。
- `_locales/`：国际化文案。
- `css/`、`img/`、`lib/`：样式、图片和第三方库资源。
- `justfile`：构建和打包脚本说明。

## 4. 推荐开发环境

- 操作系统：macOS / Linux / Windows 均可。
- 浏览器：建议 Chromium 104+；项目最低兼容 Chromium 93。
- Node.js：建议使用 LTS 版本，用于辅助脚本和构建校验。
- 可选工具：
  - `just`：用于执行 `justfile` 中的快捷任务。
  - `zip`：用于打包 ZIP。
  - `openssl`：用于生成 CRX 私钥。
  - `crx3`：用于打包 CRX。

如需准备打包环境，可参考：

```bash
npm install
npm install -g crx3
```

如果本机没有 `just`，可直接阅读并执行 `justfile` 中对应命令。

## 5. 开发与调试方式

- 日常开发优先使用“加载已解压的扩展程序”直接加载仓库目录。
- 修改 `manifest.json`、`js/`、`css/`、`html` 后，通常需要手动刷新扩展或页面。
- 调试时优先关注：
  - 扩展后台页 / Service Worker 日志
  - Popup 页面控制台
  - 目标网页控制台
  - Network 面板中的媒体请求

## 6. AI 修改代码的硬性要求

- 优先做小步、局部、可回滚的修改，不做无必要的大重构。
- 不要随意替换现有代码风格，不要为了“现代化”而整体改写。
- 不要轻易引入新依赖、新构建链、新框架。
- 不要把简单问题升级为模块化/工程化改造。
- 不要改动无关文件。
- 不要删除已有兼容逻辑，除非已经确认不再需要。
- 注释应少而准，只解释复杂逻辑，不解释显而易见的赋值。

## 7. 代码风格约束

- 保持现有项目风格，优先延续现有写法。
- 该项目历史代码混合使用 `var`、`let`、`const`、普通函数和少量箭头函数，新增代码应与所在文件风格保持一致。
- 除非有明确收益，不要在单个 PR 中顺手做格式化清洗。
- 默认使用 ASCII；除已有中文文案、注释和国际化内容外，不主动引入特殊字符。

## 8. 关键联动规则

### 8.1 修改资源捕获逻辑时

需要优先检查以下文件是否同步受影响：

- `js/background.js`
- `js/popup.js`
- `js/content-script.js`
- `js/init.js`

如果新增消息类型，必须同时检查消息发送方和接收方，不允许只改一侧。

### 8.2 修改设置项时

如果新增或调整设置项，至少检查：

- `js/init.js` 中 `G.OptionLists`
- `js/options.js` 中设置页渲染与保存逻辑
- 使用该设置的业务代码
- 必要时同步文案与默认值

### 8.3 修改脚本注入行为时

要同步检查：

- `G.scriptList`
- `catch-script/` 下对应脚本
- popup 中的触发入口
- Chrome 与 Firefox 的兼容性

### 8.4 修改下载、解析、m3u8 能力时

重点关注：

- `js/m3u8.js`
- `lib/hls.min.js`
- `lib/m3u8-decrypt.js`
- popup 到解析页的参数传递
- 请求头透传、Referer/Cookie 注入是否仍然正确

## 9. 国际化要求

- 用户可见文本原则上不要硬编码在 JS/HTML 中。
- 新增用户可见文案时，应优先补充到 `_locales/` 对应语言包。
- 至少保证英文语言包可用；如改动范围涉及核心功能，建议同步补充中文语言包。
- 不要随意修改已有 message key，避免影响旧逻辑。

## 10. Manifest 与浏览器兼容性要求

- 改动权限时必须谨慎，遵循最小权限原则。
- 修改 `manifest.json` 后，评估 `manifest.firefox.json` 是否也需要同步。
- Chrome 可用特性不代表 Firefox 直接可用，涉及以下能力时要重点验证：
  - `sidePanel`
  - `service_worker`
  - `declarativeNetRequest`
  - `scripting`
  - 下载与请求头修改

## 11. 提交前自检清单

在没有完整自动化测试的前提下，至少完成以下检查：

- 目标功能是否真的生效。
- 是否引入明显语法错误。
- 是否破坏 popup、options、background 之间的消息联动。
- 是否影响基础资源捕获。
- 如有设置项改动，是否能保存、读取、刷新后保持。
- 如有文案改动，是否存在未翻译或 key 缺失。
- 如有 manifest 改动，是否仍能正常加载扩展。

## 12. 构建与打包约定

项目当前以 `justfile` 作为构建说明，常见目标包括：

- `validate`
- `prepare`
- `build-zip`
- `build-crx`
- `build`

AI 在构建失败时应先区分：

- 是项目代码问题
- 还是本地环境缺少 `just`、`node`、`crx3`、`openssl` 等工具

不要把环境问题误判为项目问题。

## 13. 不建议 AI 做的事

- 无明确需求时批量重命名文件。
- 无明确收益时拆分大型旧文件。
- 擅自替换第三方库。
- 擅自提升最低浏览器版本要求。
- 擅自变更许可证、项目声明、免责声明。
- 为了“更优雅”修改核心抓包流程。

## 14. 推荐 AI 工作流程

1. 先读相关入口文件，确认消息流和调用链。
2. 只修改与目标相关的最小文件集合。
3. 修改后进行静态检查和最小功能验证。
4. 输出中文总结，说明改了什么、为什么这样改、还有什么风险。

## 15. 本项目的优先原则

- 兼容性优先于新潮写法。
- 可维护性优先于炫技实现。
- 最小改动优先于大规模重构。
- 可验证的修复优先于理论上的完美设计。
